# `web/src/lib/`

Runtime data layer for the portfolio site. Today this is one module (`resume.ts`)
that exposes [`data/resume.json`](../../../data/resume.json) to every Astro
component. This README documents how the mapping works so future refactors of
either the JSON or the layout don't introduce surprises.

## Contract at a glance

- **`resume.json` is the canonical source.** Profile README builder, Typst CV,
  and this site all read it. Schema lives at
  [`data/resume.schema.json`](../../../data/resume.schema.json).
- `resume.ts` does a **whole-object passthrough import** with a hand-written
  TypeScript shape — it does **not** filter, transform, or strip fields.
- Each Astro component decides which fields to render. The site only displays
  what is actually consumed; ignored fields stay in the JSON for the other
  consumers without affecting the build.

```text
data/resume.json  ──▶  web/src/lib/resume.ts  ──▶  components/sections/*.astro
                       (typed re-export, no transform)        (read what they need)
```

## How `resume.ts` is wired

```ts
import resumeData from '@data/resume.json'; // tsconfig path alias → ../data/resume.json

export type Resume = typeof resumeData;

export const resume = resumeData as unknown as {
	personalInfo: { name; title; email; location; summary };
	githubUsername: string;
	links: Link[];
	skills: SkillGroup[];
	experience: Experience[];
	education: Education[];
	certifications: Certification[];
	projects: Project[];
};
```

Two things to know:

1. **`as unknown as { … }` is a hand-written shape**, not derived from the JSON.
   It narrows `resume` for ergonomic autocompletion in components. If the JSON
   gains/renames a field, this cast does **not** error — the field is simply
   invisible to TypeScript until you add it here.
2. **`Resume = typeof resumeData`** is the *real* shape inferred by Vite's JSON
   loader and is available for tools that want the full type. Most callers use
   the curated `resume` export instead.

The `linkById(id)` helper resolves `links[]` entries by their stable `id`
(used by `Hero` for `github` and `Contact` for `email` / `linkedin`).

## What the UI actually consumes

Generated by grepping the codebase — keep this table updated when you add a section
or change which fields a section reads.

| Source field                        | Read by                                        | Notes                                       |
| ----------------------------------- | ---------------------------------------------- | ------------------------------------------- |
| `personalInfo.name`                 | `Header`, `Footer`, `Hero`, `BaseLayout`       | Header lowercases & dot-joins (e.g. `carlos.ferreyra`) |
| `personalInfo.title`                | `Hero`, `BaseLayout`                           | Used in `<title>` and og tags               |
| `personalInfo.email`                | `About`                                        | Rendered as `mailto:` link                  |
| `personalInfo.location`             | `About`                                        |                                             |
| `personalInfo.summary`              | `Hero`, `About`, `BaseLayout`                  | Default meta description                    |
| `githubUsername`                    | `Hero`                                         | Terminal-prompt cue (`~ /<username> $`)     |
| `links[]` (full array)              | `Footer`, `Contact`                            | `Contact` uses `linkById` to pin email + linkedin first |
| `skills[]` (full array)             | `Skills`                                       | Renders every group + every item            |
| `experience[]` (full array)         | `Experience`                                   | Vertical timeline                           |
| `education[0]` (first only)         | `About`                                        | **Only the first entry is shown** — see Limitations |
| `certifications[]` (full array)     | `Certifications`                               | Linked grid                                 |
| `projects[]` (full array)           | `Projects` → `ProjectCard`                     | Optional `thumbnail` + `demo` fields render when present |

### Fields kept in the JSON but ignored by the site

These exist for other consumers (the Typst CV builder, the README templater) and
have no effect on the portfolio:

- `summaries.{devops,backend,fullstack,cli}` — variant-specific summary copy
- `variants[]` — Typst CV section toggles
- `tags[]` on `skills`, `experience`, `education`, `certifications`, `projects`
  — currently displayed only on `ProjectCard`; no filter UI yet

## Mapping rules — what happens when you edit `data/resume.json`

| Edit                                                       | Effect on the site                                      |
| ---------------------------------------------------------- | ------------------------------------------------------- |
| Change a value of a field the UI reads                     | Flows through on next build, no code change             |
| Add an entry to any `*[]` array (`projects`, `skills`, …)  | Renders automatically — every section uses `.map()`     |
| Add a **second** entry to `education`                      | Won't appear — `About.astro` reads `education[0]` only  |
| Add a brand-new top-level field                            | Invisible until a component imports it (see "Add a new field" below) |
| Rename a field that the UI reads                           | Build fails on `astro check` — caught before deploy     |
| Remove a field that the UI reads                           | Build fails on `astro check` — caught before deploy     |
| Edit `summaries`, `variants`, or any `tags[]`              | No visible effect on the site (other consumers only)    |
| Add `projects[].thumbnail` (optional)                      | Image renders in `ProjectCard`; absent → monospace fallback |
| Add `projects[].demo` (optional)                           | "Demo ↗" link appears next to the source link           |

## Add a new field — checklist

Suppose you want to expose `interests[]` (a new top-level array of strings).

1. Add the field to `data/resume.json`.
2. Update `data/resume.schema.json` so the README/Typst tools stay valid.
3. In `web/src/lib/resume.ts`:
   - Either declare a new `type Interest = …` and add `interests: Interest[]`
     to the `resume` cast object, or rely on `Resume = typeof resumeData`.
4. Add a section component under `web/src/components/sections/` that imports
   `resume` and `.map()`s over `resume.interests`.
5. Mount it in `src/pages/index.astro` and `src/pages/es/index.astro`.
6. Add localized strings to `src/i18n/{en,es}.ts` and the `Dictionary` type in
   `src/i18n/types.ts`.
7. Run `bun run build` — `astro check` validates the type chain end to end.

## Limitations & known sharp edges

- **`education[0]` only.** The About section assumes a single education entry.
  If you add more, either fan them out in `About.astro` or pick the most recent
  programmatically.
- **No transform layer.** Sorting (e.g. certifications by `year` desc) currently
  happens in the order they appear in the JSON. If you need ordering rules, add
  selectors to this `lib/` module rather than sorting inline in components.
- **No filter by variant.** The site renders the *full* dataset; Typst CV
  variants (`devops`, `backend`, `fullstack`, `cli`) are deliberately ignored.
  If you want, e.g. `/devops` to show only DevOps-tagged entries, add helpers
  here (`projectsForVariant(name)`, `skillsForVariant(name)`) and call them
  from per-route pages.
- **Hand-written shape can drift.** Because `resume` is cast (not inferred), a
  field rename in the JSON won't error here unless something else reads it.
  If you care, switch to `typeof resumeData` and let Vite's loader own the type.

## Future swap to a remote fetch

When [Phase 3 of the roadmap](../../../ROADMAP.md) publishes `resume.json` as
the canonical remote artifact, replace the static import with a build-time
fetch — no other module changes:

```ts
// before
import resumeData from '@data/resume.json';

// after — fetched at build time, never at runtime (the site stays static)
const resumeData = await fetch(
	'https://raw.githubusercontent.com/carlosferreyra/carlosferreyra/main/data/resume.json',
).then((r) => r.json());
```

Components keep importing `resume` from this module — the contract is stable.
